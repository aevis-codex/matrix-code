# MatrixCode 设计规格

## 概述

MatrixCode 是一个面向多人实时协作的软件交付平台，用 AI 辅助团队完成从需求到演示部署的完整流程。它不是聊天壳、不是 Jira 仿制品，也不是外部工具索引。产品、开发、测试、运维的主要工作都在同一个桌面工作台内完成，并连接到独立的 Java 团队服务器。人类成员负责审核、授权、验收和处理模糊决策；各角色智能体负责生成文档、修改代码、运行测试、准备环境，并推动工作进入可审计的交付状态流。

第一版支持 macOS 和 Windows 桌面客户端，配套独立安装的 Spring Boot 团队服务器，并以 Docker Compose 为演示环境和测试环境的主要部署目标。系统需要为未来生产部署和团队规模扩展预留空间，但第一版不做完整的云平台或 Kubernetes 自动化部署。

## 目标

- 让团队在 MatrixCode 内完成从需求描述到演示部署的完整闭环。
- 将产品、开发、测试、运维作为一等角色，每个角色都可以配置模型和工具。
- 让每次角色交接都有文档、有版本、有状态、有审计记录。
- 支持在平台内完成本地工作区编码、测试执行、Docker 环境搭建、SSH 配置和部署动作。
- 使用严格上下文门禁，提高模型供应商的提示词缓存（prompt cache）命中率，降低 token 成本。
- 提供清爽、克制、信息密度高的桌面 UI，避免明显的「AI 炫技感」。

## 第一版非目标

- 完整 SaaS 多租户计费。
- Kubernetes、Vercel、Fly、Render 或真实生产环境的全自动部署。
- 无人工确认的生产 SSH 自动部署。
- 完全替代 Git 托管平台或 CI 系统。
- 脱离智能体交付流程，单独构建通用 Jira 克隆产品。

## 产品流程

项目负责人创建项目组，邀请成员，分配产品、开发、测试、运维角色，并为每个角色配置默认模型。项目可以覆盖组织级默认模型。

产品成员在平台内通过对话描述需求。产品智能体生成 PRD、用户流程、初始界面说明、验收标准和待确认问题。产品成员审核结果、回答问题、要求修改，并冻结通过审核的版本。开发、测试、运维默认只能消费冻结版本。

开发成员从桌面客户端选择一个或多个本地工作区。开发智能体读取冻结 PRD 和验收标准，搭建或更新项目，编写代码，执行自测，生成数据库脚本、接口文档和部署文档。所有文件变更都会形成 diff，并关联到具体任务。

运维成员根据部署文档创建 Docker Compose 测试环境。平台允许配置 SSH 目标和凭证，但 SSH、私钥使用、远程变更、部署、回滚都必须经过人工授权。已授权沙箱内的 Docker Compose 测试环境操作可以自动执行。

测试成员通过对话生成并执行测试用例。测试可以通过对话记录 Bug、附加日志或截图、流转 Bug 状态、要求开发修复，并执行回归测试。测试失败会把工作退回开发；测试通过后生成测试报告，并提交给产品验收。

产品成员进行最终验收。验收通过后，运维执行演示部署，或将 Docker Compose 环境标记为可发布状态。部署失败时，系统创建事故记录，并给出回滚建议。

## 必要交接产物

每次角色交接都必须包含明确产物：

- 产品交给开发：冻结版 PRD、用户流程、验收标准、需求版本号。
- 开发交给测试：代码 diff、自测结果、接口文档、数据库脚本、实现说明。
- 开发交给运维：部署文档、环境变量、服务拓扑、健康检查、回滚说明。
- 运维交给测试：环境地址、部署日志、容器状态、健康检查结果。
- 测试交给产品：测试报告、通过/失败用例、Bug 列表、未解决风险、回归结果。
- 产品交给运维：验收通过记录、发布授权。
- 运维交给项目组：部署记录、运行状态、回滚或事故状态。

## 状态模型

主要文档和任务状态：

```text
草稿 -> 待审核 -> 已冻结 -> 执行中 -> 待验收 -> 已完成
```

Bug 状态：

```text
新建 -> 已确认 -> 修复中 -> 待回归 -> 已关闭 -> 重新打开
```

需求变更不能覆盖已冻结 PRD。需求变更必须创建变更单，引用被冻结的版本，生成新的候选版本，并经过审核后才能再次冻结。测试失败会将相关任务退回开发。验收失败会将任务退回对应角色节点。部署失败会创建运维事故记录和回滚建议。

所有状态变化都以追加事件记录。当前状态是事件流的投影，不是唯一事实来源。

## 架构

第一版采用模块化单体团队服务器、桌面客户端和本地执行代理。

推荐技术栈：

- 服务端：Java 21、Spring Boot 3、PostgreSQL、Redis、WebSocket/SSE。
- 桌面端：Tauri 2、React、TypeScript。
- 本地执行：随桌面端打包的执行代理进程。
- 部署目标：本地或内网测试/演示环境的 Docker Compose。

Spring Boot 服务端负责身份、项目组、文档、工作流状态、角色会话、模型编排、审计、成本统计和实时同步。服务端不直接访问用户本地文件。

桌面客户端负责工作台 UI、本地工作区选择、本地权限提示、终端/SSH 授权入口、实时更新和文件系统集成。

执行代理运行在用户机器上，只能访问被明确授权的工作区。它负责文件读写、命令执行、Git diff 采集、测试执行、Docker Compose 操作、日志采集和已授权的 SSH 动作，并将结构化结果返回服务端。

## 安装与分发

macOS 客户端以签名 `.dmg` 或 `.pkg` 形式发布。Windows 客户端以签名 `.msi` 或 `.exe` 安装包发布。两个平台的客户端都包含桌面 UI 和本地执行代理。

团队服务器提供两种安装方式：

- 推荐方式：Docker Compose 包，包含服务端、PostgreSQL、Redis 和持久化卷。
- 进阶方式：Java 可执行包，适合已经自管 PostgreSQL 和 Redis 的团队。

首次启动向导负责连接团队服务器、登录账号、检查本地执行代理健康状态，并帮助配置第一个组织、项目模板和模型供应商。

## 服务端模块

- 身份与团队：用户、组织、角色、邀请、会话和在线状态。
- 项目工作流：项目、阶段、任务、Bug、审批、状态流转和事件投影。
- 文档中心：版本化文档、冻结版本、diff、引用关系、产物元数据和搜索索引。
- 智能体编排：角色智能体、会话生命周期、工具调度、审批、重试和交接产物生成。
- 模型网关：供应商配置、模型选择、流式响应、用量统计、费用计算和缓存指标。
- 上下文引擎：严格上下文组装、文档门禁、检索、摘要、压缩和角色记忆。
- 执行网关：与桌面执行代理建立安全通信，并规范化执行结果。
- 审批与审计：审批规则、危险动作门禁、不可变事件日志和审核轨迹。
- 实时同步：通过 WebSocket/SSE 推送工作流更新、日志、审批和模型进度。
- 插件与技能运行时：工具注册、技能发现、按阶段暴露工具、区分只读工具和写入工具。
- 组织策略：模型白名单、费用预算、SSH 策略、命令白名单、工作区规则和审计保留周期。
- 项目模板：Java 后端、前后端分离、Spring Cloud 和空白项目模板。

## 协作模型

平台默认实时协作。用户无需手动刷新，即可看到工作流状态、智能体进度、日志、审批、文档变更、评论和 Bug 更新。

文档支持版本历史、冻结版本、评论、成员提及和审核状态。草稿文档采用乐观并发和冲突检测。冻结文档会生成不可变版本，下游角色通过版本 ID 引用它。

任务、Bug、文档、审批和部署都有评论线程。智能体生成的变更必须清晰标记为已生成、已审核、已接受、已拒绝或已被替代。

工作流看板是追加事件的投影。重建投影时，同一份事件流必须能复现同样的可见状态。

## 角色智能体

每个角色都有独立的模型配置、系统提示词、工具策略、记忆范围和会话历史。

产品智能体：

- 输入：用户需求对话、组织规则、项目模板、必要的历史冻结文档。
- 工具：文档生成器、界面说明生成器、验收标准生成器、变更单创建器、文档审查器。
- 输出：PRD、用户流程、初始界面说明、验收标准、变更单。

开发智能体：

- 输入：冻结 PRD、验收标准、已选择工作区元数据、相关 Bug 报告、编码规范。
- 工具：工作区文件工具、命令/测试工具、diff 生成器、数据库脚本生成器、接口文档生成器。
- 输出：代码变更、自测结果、数据库脚本、接口文档、实现说明。

测试智能体：

- 输入：冻结需求、接口文档、部署状态、Bug 历史、测试报告。
- 工具：测试用例生成器、HTTP/API 测试运行器、可用时的浏览器测试运行器、Bug 记录器、回归测试运行器。
- 输出：测试用例、Bug 报告、回归结果、最终测试报告。

运维智能体：

- 输入：部署文档、环境策略、Docker/SSH 配置、运行日志。
- 工具：Docker Compose 运行器、环境校验器、健康检查器、日志采集器、带审批门禁的 SSH 运行器。
- 输出：环境记录、部署日志、健康检查结果、回滚建议、事故记录。

## 模型与插件设计

模型网关必须通过接口优先的注册机制支持 OpenAI 兼容接口、DeepSeek、Qwen 和 Anthropic 风格的供应商。新增模型通常应该是配置变更，而不是代码变更。

插件和技能以注册工具的形式暴露，不写死在角色逻辑中。角色只能获得当前阶段允许的工具。只读工具可以更积极地并行和自动审批。写入、命令、SSH、部署、凭证类工具必须经过审批门禁。

模型供应商密钥在服务端加密存储，或通过服务端环境变量提供。除非组织策略明确允许个人密钥，否则桌面客户端不需要向用户暴露组织级供应商密钥。

## 上下文与 token 成本设计

MatrixCode 将缓存命中率和低 token 成本视为产品行为。

设计参考 Reasonix 的关键思路：

- 在角色会话内保持系统提示词、角色契约、组织规则、工具 schema 和稳定项目规则的字节级稳定。
- 使用追加事件历史，不重写历史消息。
- 产品、开发、测试、运维使用独立角色会话，避免模型切换或角色上下文扰动其他角色的缓存前缀。
- 角色交接通过冻结文档、摘要、diff 和状态事件完成，而不是传递完整聊天记录。
- 通过只读工具按需检索历史对话、日志、文档和产物。
- 只有在上下文压力超过阈值时，才低频、显式地进行压缩。
- 对提示词、工具 schema、上下文组装和摘要策略的缓存敏感变更进行记录和审查。

默认采用严格上下文门禁：

- 产品对话不会整体流入开发。开发只接收冻结 PRD、验收标准、明确决策和关联变更单。
- 开发日志不会整体流入测试。测试只接收实现说明、接口文档、自测结果和相关 diff。
- 测试执行历史不会整体流入产品。产品只接收测试报告、未解决风险摘要和关联 Bug 状态。
- 运维日志默认以摘要形式流转，除非角色明确检索某段日志窗口。

上下文引擎按顺序组装角色请求：

1. 稳定系统前缀：平台规则、角色契约、输出契约、工具契约。
2. 稳定项目前缀：冻结项目规则、编码规范、组织策略。
3. 冻结产物：当前阶段需要的 PRD、接口文档、数据库脚本、部署文档或测试报告。
4. 近期状态事件：仅包含当前任务、Bug、审批和环境状态。
5. 检索证据：来自文件、日志、历史对话或文档的有界片段。
6. 用户本轮指令：当前输入。

UI 需要展示上下文窗口占用、缓存命中 token、缓存未命中 token、费用、压缩状态，以及本次请求进入上下文的产物列表。

## 权限与安全

默认模式是：沙箱内自动执行，危险动作必须确认。

项目或工作区授权后可自动执行：

- 读取已授权工作区内的文件。
- 在已授权工作区内，围绕当前任务写入文件。
- 运行已配置的构建和测试命令。
- 生成 diff、文档、数据库脚本和报告。
- 启动或停止当前项目的 Docker Compose 测试环境。

必须人工确认的动作：

- 批量删除文件。
- 写入已授权工作区以外的路径。
- 安装系统级软件。
- 会发送凭证或敏感文件的网络动作。
- SSH 连接、私钥使用、远程文件变更、远程服务重启。
- 部署、回滚、类生产环境变更。
- 持久化新密钥或修改组织策略。

每次工具调用都要记录操作者、角色、模型、工具、参数摘要、审批决策、结果、时间戳、工作区和关联任务/产物。敏感值必须脱敏。

## 可靠性与可观测性

管理员可以查看服务端健康状态、执行代理心跳、模型供应商故障、队列深度、WebSocket 连接数，以及 token 和费用指标。

长时间运行的智能体任务必须持续发送进度事件。任务可以暂停、取消、重试，或从最近的持久状态恢复。失败的工具调用需要保留足够上下文，方便所属角色重试或交接。

执行代理连接必须绑定到已签名的桌面会话和项目成员身份。如果桌面客户端在本地任务执行期间断开连接，服务端需要把任务标记为已中断，并保留所有已完成产物和日志。

## UI 设计

桌面 UI 应该像一个严肃的交付工作台：克制、紧凑、可读、偏运营工具感。避免通用 AI 渐变、营销式首页、过大的卡片和无意义装饰。

主布局：

- 左侧栏：组织、项目组、角色切换、当前工作区、在线成员。
- 主工作区：当前角色任务、对话、文档预览/编辑器、diff 视图、测试/部署日志、审批卡片。
- 右侧栏：上下文和费用指标、产物、当前审批、缓存状态、工作流状态。
- 底部状态栏：当前模型、角色、工作区、执行模式、费用、缓存命中率和连接状态。

关键界面：

- 产品需求录入与 PRD 审核。
- 开发编码运行界面，包含 diff 和测试输出。
- 测试 Bug 记录与回归流程。
- 运维环境和部署控制台。
- 文档版本与冻结视图。
- 包含任务和 Bug 状态的工作流看板。
- 审批收件箱。
- 模型和组织策略设置。

## MVP 范围

第一版需要包含：

- 用户登录和团队服务器连接。
- 组织、项目、成员、角色管理。
- 按角色配置模型。
- 项目创建，至少支持空白模板和 Spring Boot 模板。
- 产品需求对话生成 PRD 和验收标准。
- 文档中心支持版本、冻结、diff 和引用关系。
- 工作流状态机和 Bug 状态机。
- 桌面端选择本地工作区，并与执行代理配对。
- 开发智能体支持文件编辑、命令执行、测试、diff、数据库脚本文档和接口文档。
- 运维通过 Docker Compose 搭建测试环境。
- 测试支持测试用例生成、Bug 记录、状态流转和测试报告。
- 产品验收和运维演示部署记录。
- WebSocket/SSE 实时更新。
- 用量、缓存和费用仪表盘。
- 审批门禁和审计日志。

## 后续范围

- 超出演示部署的生产 SSH 部署剧本。
- Kubernetes 和云平台部署目标。
- 大规模浏览器自动化 UI 测试。
- CI 供应商集成。
- GitHub/GitLab Pull Request 集成。
- 企业级细粒度 RBAC 和 SSO。
- 多租户 SaaS 计费。
- 市场化插件分发。

## 验收标准

- 产品用户可以创建项目、描述需求、审核生成的 PRD 内容，并冻结版本。
- 开发用户可以选择本地工作区，并让开发智能体根据冻结 PRD 创建或修改项目文件。
- 系统记录每个生成产物和状态变更。
- 开发智能体可以运行已配置测试，并把结果附加到任务。
- 运维可以根据部署文档创建 Docker Compose 测试环境。
- 测试可以通过对话创建 Bug，在 Bug 状态机中流转，并生成测试报告。
- 产品可以接受或拒绝测试结果。
- 运维可以在验收后创建演示部署记录。
- 危险动作必须审批，已授权沙箱操作可以自动执行。
- 角色请求可以展示上下文块、token 用量、缓存命中/未命中和预估费用。
- 两个连接到同一项目的用户可以实时看到工作流和产物更新。

## MVP 实现决策

- API 形态：命令和查询接口使用 REST，实时流使用 WebSocket/SSE。
- 执行代理打包方式：作为 Tauri sidecar 随桌面客户端打包。

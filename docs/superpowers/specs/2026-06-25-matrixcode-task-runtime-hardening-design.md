# MatrixCode 第八阶段本地任务运行态加固设计规格

## 背景

第七阶段已经把可执行本地命令升级为长任务队列，任务可以入队、运行、追加日志、取消，并展示在工作台和桌面端右侧指标栏。当前还剩两个影响后续阶段稳定性的运行态问题：

- 桌面端只在用户动作完成后刷新工作台。任务自然完成、持续输出日志或状态变化时，用户需要手动触发其他动作才能看到最新状态。
- 服务端 `acceptedTasks` 用于防止同一任务编号重复调度，但当前无上限保留。长时间运行后会积累已经结束且已被任务历史淘汰的任务编号。

第八阶段不扩展执行能力，而是加固第七阶段的可见性和内存边界，为后续 Docker Compose 生命周期、远程部署审批和日志流直连做准备。

## 范围

第八阶段实现以下能力：

- 桌面端在存在 `QUEUED` 或 `RUNNING` 本地任务时，自动轻量刷新工作台。
- 自动刷新不覆盖当前工作台页面，只更新右侧任务状态、日志和阶段指标。
- 当运行中任务结束或刷新失败时，轮询停止或保留当前可见数据并展示已有同步错误。
- 服务端本地任务队列的已接收任务标记使用有界保留策略，防止 tombstone 无界增长。
- 已接收任务标记裁剪后，仍不得重复执行还在任务历史内的同一任务编号。
- 文档补充第八阶段验证路径，并收拢早期 MVP 计划中已复验通过的收尾项。

第八阶段不实现以下内容：

- WebSocket 或 SSE 直连日志流。
- Docker Compose 生命周期管理和容器日志采集。
- 真实 SSH、远程部署、凭证读取或生产环境连接。
- 数据库持久化、任务重试、优先级队列和并发池配置 UI。
- 登录鉴权、多成员权限和组织策略。

## 用户体验

用户批准一条耗时命令后，右侧“本地执行代理”卡片会持续显示任务状态和最近日志。任务运行期间无需手动刷新页面，即可看到 `排队中`、`运行中`、`成功`、`失败` 或 `已取消` 的变化。

自动刷新保持轻量：

- 只在存在 `QUEUED` 或 `RUNNING` 任务时启动。
- 每 2 秒刷新一次工作台。
- 刷新期间顶部状态继续显示 `刷新中`。
- 刷新失败时保留当前数据，避免把用户带回错误页。

## 服务端设计

`LocalTaskQueueService` 保留两个去重结构：

- `scheduledTasks`：短期调度锁，只保护正在提交或已经提交到执行器但尚未进入运行函数的任务。
- `acceptedTasks`：已接收任务编号 tombstone，用于阻止任务历史被淘汰后的重复执行。

第八阶段把 `acceptedTasks` 从无界 `ConcurrentHashMap` 改为有界保留结构：

- 默认最多保留 200 个任务编号。
- 新任务接收成功后追加到保留队列。
- 超过上限时删除最旧任务编号。
- `shutdown()` 清空保留结构。
- 任务历史仍由 `LocalTaskStore` 保留最近 20 个终态任务和所有活跃任务。

重复执行保护分两层：

- 如果任务还在 `LocalTaskStore` 历史内，`recordNew` 或冲突检查会拒绝重复编号。
- 如果任务已经被历史淘汰，`acceptedTasks` 在保留窗口内继续拒绝重复编号。

## 桌面端设计

`App` 增加运行中任务检测：

- 从 `workbench.localExecution.activeTasks` 中查找 `QUEUED` 或 `RUNNING`。
- 存在运行态任务时启动 `setInterval`，每 2 秒调用 `refreshWorkbench({ keepCurrent: true })`。
- 当工作台没有运行态任务时清理定时器。
- 组件卸载时清理定时器。

审批、取消、健康检查等已有动作仍保留动作完成后的立即刷新。自动刷新只补齐任务自然变化路径。

## 测试策略

服务端：

- 单元测试验证已接收任务标记超过上限后会裁剪最旧记录。
- 单元测试验证裁剪不会影响任务历史内重复编号拒绝。

桌面端：

- Vitest 使用 fake timers，验证运行中任务会触发自动刷新。
- 验证任务刷新到终态后不再继续轮询。
- 验证没有运行态任务时不会启动额外刷新。

文档：

- 本地运行指南新增第八阶段验证说明。
- 阶段计划记录服务端、桌面端、构建和运行态验证结果。

## 验收标准

- 服务端相关测试和服务端全量测试通过。
- 桌面端测试和构建通过。
- 桌面端运行中任务可以自动刷新为终态。
- `acceptedTasks` 保留数量有上限，长时间运行不会无界增长。
- 文档中没有占位符、英文模板标题或不符合本地 Maven 路径的命令。

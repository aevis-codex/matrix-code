# MatrixCode Agent 运行审计详情设计

## 背景

第 58 阶段已经让运行中心可以按 Agent 事件类型筛选，并展示项目级最近模型请求线索。下一步需要把同一运行的关键信息聚合成一个可读审计摘要，让操作者不用在右侧指标、模型网关、事件时间线之间来回拼接。

## 目标

在运行中心新增“Agent 审计详情”卡片，展示最新 Agent 运行的审计摘要：运行身份、角色、操作者、模型、目标、结果、恢复策略、事件计数和最近模型请求缓存线索。

## 推荐方案

- 继续复用桌面端已加载的 `agentRuns`、`agentRunEvents` 和 `modelGateway.recentRequests`。
- 在 `OperationsDetailPanel` 中派生最新运行和最近模型请求的审计摘要。
- 审计详情卡片展示：
  - `运行 run-1 · 失败 · 可重试`
  - `目标 修复支付失败重试`
  - `操作者 user-dev · 角色 开发`
  - `模型 deepseek/deepseek-chat · coding`
  - `工具事件 1 · 失败事件 1`
  - `恢复 run-0 · 可重试`
  - `摘要 测试命令超时，未产生交接文档`
  - `失败 测试命令超时`
  - `模型请求 request-1 · PROVIDER · prefix fp-cache-001`

## 替代方案与取舍

- 方案 A：前端聚合已有数据。成本最低，不新增 API 和 DDL，适合当前阶段。缺点是模型请求仍是项目级最近线索，不是强关联。
- 方案 B：新增后端审计详情 API。数据语义更强，但需要 DTO、服务编排和测试扩大，当前已有数据足够支撑排障摘要。
- 方案 C：新增 Agent 运行与模型请求关联表。适合后续强审计，但当前还没有真实业务动作写入关联 ID，过早建表会增加迁移风险。

推荐方案 A。它能立即提升可用性，并保持第 58 阶段建立的轻量前端路径。

## 边界

- 本阶段不新增后端 API、不新增 DDL、不改变 Agent Runtime 写入链路。
- 最近模型请求仍标注为线索，不作为强关联审计证据。
- 不展示完整 prompt、模型响应、工具输出、文件内容、API Key、数据库密码或异常堆栈全文。
- 审计详情只进入运行中心，不放回右侧指标，避免右侧区域重新变得拥挤。

## 验收标准

- 运行中心出现 `Agent 审计详情` 卡片。
- 卡片展示运行 ID、失败状态、可重试状态、目标、操作者、角色、模型和 Agent 类型。
- 卡片展示工具事件数和失败事件数。
- 卡片展示恢复来源、摘要和失败摘要。
- 卡片展示最近模型请求 ID、缓存来源和 prefix fingerprint。
- 桌面端目标测试先红后绿；桌面端全量、桌面构建、服务端全量、真实集成、静态检查和密钥扫描继续通过。

# MatrixCode 桌面端编码智能体执行准备入口设计

## 背景

第 24 阶段已经在服务端提供编码智能体执行准备桥接：开发角色可以基于目标、授权工作区和测试命令生成执行计划，服务端负责校验工作区、采集 Git diff 基线、把测试命令提交到本地执行审批流，并把代码编辑步骤标记为需要审批。桌面端当前只有开发交付表单，无法从开发工作区直接触发这条真实执行准备链路。

## 目标

在开发角色工作区增加“编码智能体执行准备”入口，让用户可以选择已授权工作区、填写开发目标和测试命令，调用真实服务端接口生成执行准备报告，并在当前工作区展示步骤、审批状态、测试任务和 diff 基线。

## 非目标

- 不在桌面端直接执行代码修改。
- 不绕过本地执行审批策略。
- 不实现自动 patch、自动提交或远程部署。
- 不在前端持久化 API key、数据库密码或模型密钥。

## 推荐方案

采用“面板内入口 + 结果摘要”的方式：

1. `desktop/src/api/client.ts` 新增编码智能体执行准备类型和 `prepareCodingAgentExecution` API 封装。
2. `DeveloperPanel` 增加执行准备表单，使用工作台已有的授权工作区作为下拉选项。
3. `App.tsx` 负责把项目 ID、开发角色路径 `developer` 和固定前端操作者 `user-dev` 传给 API。
4. 执行准备成功后在开发面板内展示任务状态、测试命令状态、diff 摘要和执行步骤。

## 交互设计

- 开发工作区顶部新增一个紧凑区块：“编码智能体执行准备”。
- 无授权工作区时，表单禁用并提示先到运行中心授权工作区。
- 默认测试命令为 `git status`，用于安全地验证本地执行链路；用户可改成项目测试命令。
- 生成后展示：
  - 目标任务状态。
  - 测试命令任务状态。
  - Git diff 是否为仓库、变更文件数量和 stat。
  - 按顺序列出上下文召回、计划审查、文件阅读、代码编辑、测试命令、diff 审查、交付回溯等步骤。

## 数据契约

前端请求：

```json
{
  "goal": "修复支付失败重试",
  "workspaceId": "workspace-1",
  "actorId": "user-dev",
  "testCommand": "git status"
}
```

接口路径：

```text
POST /api/projects/{projectId}/roles/{role}/coding-agent/execution-plans
```

前端响应类型对齐服务端：

- `CodingAgentExecutionPlan`
- `CodingAgentTask`
- `CodingAgentExecutionStep`
- `CodingAgentExecutionStatus`
- `CodingAgentStepType`

## 错误处理

- 服务端业务错误直接显示中文错误消息。
- 网络错误沿用 API client 的“团队服务器连接失败”。
- 未选择工作区或目标为空时在面板内显示校验错误。
- 执行准备失败不清空用户输入。

## 测试策略

- API client 红绿测试：验证请求路径、POST body 和 JSON header。
- UI 红绿测试：验证开发角色可以从面板触发执行准备，API 参数正确，成功后显示审批状态、步骤和 diff 摘要。
- 回归验证：桌面端测试、桌面端构建、必要的浏览器手动验收。

## 验收标准

- 开发角色工作区可以真实调用服务端执行准备接口。
- UI 不把执行准备内容放到右侧运行指标区域。
- 代码编辑步骤明确显示需要审批。
- 测试命令进入本地执行审批/任务状态展示。
- 无授权工作区时不会发起请求。
- 本阶段变更不引入密钥明文。

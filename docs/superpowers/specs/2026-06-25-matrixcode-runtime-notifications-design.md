# MatrixCode 第十一阶段运行态提醒中心设计规格

## 背景

第十阶段已经让桌面端订阅项目 SSE 事件流，并在本地任务和 Compose 运行态变化时自动刷新工作台。当前用户仍需要主动看右侧卡片或事件流，才能发现待审批命令、任务完成、任务失败或 Compose 失败。对于一个多角色工作台来说，这些运行态变化已经具备“提醒”语义：它们不只是数据刷新，还需要把用户注意力引到下一步动作。

第十一阶段目标是在不扩大后端协议复杂度的前提下，把现有运行态事件转化为桌面端内的提醒中心。提醒中心只做应用内提示和待办计数，不接入系统级通知、不做数据库持久化，也不引入多成员在线状态。

## 目标

- 桌面端收到待审批、本地任务终态或 Compose 失败事件后，生成应用内提醒。
- 左侧角色导航或右侧指标区展示待审批命令数量，让用户不用寻找卡片内容就能知道有动作需要处理。
- 工作台顶部展示最近一条非阻断提醒，用户可以关闭。
- 右侧指标栏新增“运行态提醒”列表，显示最近提醒、时间和等级。
- 提醒来自现有工作台快照和 SSE 事件，不要求服务端新增提醒表。
- 测试覆盖提醒生成、提醒关闭、待审批计数和事件刷新后的提醒同步。

## 非目标

- 不接入 macOS、Windows 或浏览器系统通知。
- 不持久化提醒，不跨刷新保留已关闭状态。
- 不新增登录、用户偏好或多成员权限。
- 不把所有项目事件都做成提醒；只处理运行态和审批相关事件。
- 不实现逐行日志流，也不替代第十阶段的工作台快照刷新。

## 推荐方案

采用“桌面端派生提醒状态”的方案：

- 服务端保持现状，不新增提醒领域对象。
- 桌面端从 `ProjectWorkbench` 快照派生待审批计数和最近运行态提醒。
- SSE 事件到达后仍先刷新工作台，刷新完成后再从新快照派生提醒。
- App 内维护用户关闭的提醒 ID 集合，避免当前会话里反复弹出同一条顶部提醒。

备选方案一是在服务端新增提醒聚合和接口，优点是后续可持久化，缺点是第十一阶段会牵动数据库前置设计。备选方案二是直接使用浏览器 Notification API，用户感知强，但需要权限提示、系统差异和隐私说明。当前推荐方案改动最小，能直接利用第十阶段 SSE，风险最低。

## 提醒范围

第十一阶段生成以下提醒：

- `APPROVAL_REQUIRED`：存在 `APPROVAL_PENDING` 本地命令，需要用户批准或拒绝。
- `LOCAL_TASK_SUCCEEDED`：最近本地命令执行成功。
- `LOCAL_TASK_FAILED`：最近本地命令执行失败。
- `LOCAL_TASK_CANCELED`：最近本地命令被取消。
- `COMPOSE_FAILED`：最近 Compose 操作失败。
- `COMPOSE_SUCCEEDED`：最近 Compose 操作成功。

提醒等级：

- `ACTION`：需要用户处理，例如待审批。
- `SUCCESS`：任务或 Compose 动作成功。
- `WARNING`：任务取消。
- `ERROR`：本地任务失败或 Compose 失败。

每条提醒使用稳定 ID，避免同一个快照反复生成不同提醒：

- 待审批：`approval:<taskId>`
- 本地任务终态：`local-task:<taskId>:<status>`
- Compose 操作：`compose:<operationId>:<status>`

## 桌面端设计

新增 `desktop/src/runtimeNotifications.ts`，集中放置纯函数：

- `RuntimeNotification` 类型。
- `deriveRuntimeNotifications(workbench)`：从工作台快照派生提醒，按发生时间倒序返回最多 5 条。
- `countPendingApprovals(workbench)`：统计 `APPROVAL_PENDING` 任务数量。
- `latestVisibleNotification(notifications, dismissedIds)`：返回顶部提醒候选。

`desktop/src/App.tsx` 负责：

- 在 ready 状态下计算提醒和待审批数量。
- 保存当前会话已关闭提醒 ID。
- 在顶部阶段栏下方显示最近一条未关闭提醒。
- 把待审批数量传给 `RoleSwitcher` 和 `InspectorPanel`。
- 把提醒列表传给 `InspectorPanel`。

`desktop/src/components/RoleSwitcher.tsx` 负责：

- 运维角色旁显示待审批徽标。
- 徽标文本为数字，最多显示 `9+`。

`desktop/src/components/InspectorPanel.tsx` 负责：

- 在“本地执行代理”卡片保留现有审批按钮。
- 新增“运行态提醒”卡片，列出最近 5 条提醒。
- 不在卡片里提供复杂操作入口，只展示摘要和时间。

`desktop/src/App.css` 增加：

- 顶部提醒条样式。
- 角色徽标样式。
- 提醒列表等级样式。

## 数据流

1. App 首次加载工作台。
2. `deriveRuntimeNotifications` 从快照生成提醒。
3. App 根据 `dismissedNotificationIds` 选择顶部提醒。
4. 用户关闭顶部提醒后，提醒 ID 进入当前会话关闭集合。
5. SSE 事件到达后，App 刷新工作台。
6. 新快照生成新提醒；如果 ID 不在关闭集合中，顶部提醒再次出现。

## 错误处理

- 提醒派生函数必须容忍空数组、缺失最新操作和无效时间。
- SSE 连接失败时不生成错误提醒，继续依赖第十阶段 fallback。
- 工作台刷新失败时不新增提醒，保留当前快照和第十阶段同步失败提示。
- 已关闭提醒只在当前浏览器会话内生效，刷新页面后可以重新显示。

## 测试策略

- 纯函数测试：覆盖待审批、成功、失败、取消、Compose 成功、Compose 失败、数量上限和排序。
- App 测试：初始快照有待审批任务时展示顶部提醒和运维徽标。
- App 测试：关闭顶部提醒后，当前提醒不再显示。
- App 测试：收到本地任务完成事件并刷新工作台后，显示成功提醒。
- Inspector 测试：右侧“运行态提醒”卡片展示等级、摘要和时间。
- 全量验证：服务端测试、桌面测试、桌面 build、Tauri 命令入口、浏览器提醒验证。

## 验收标准

- 有待审批命令时，运维角色能看到待审批数量徽标。
- 有待审批命令时，顶部出现可关闭的“需要审批”提醒。
- 本地任务成功、失败或取消后，顶部和右侧提醒列表能显示对应提醒。
- Compose 成功或失败后，右侧提醒列表能显示对应提醒；失败为错误等级。
- 关闭提醒不会影响实际审批、取消或刷新能力。
- 所有新增和既有测试通过，文档包含第十一阶段验证路径。

## 后续阶段入口

第十二阶段可以继续扩展以下方向：

- 将提醒持久化到服务端，并记录已读状态。
- 接入系统级通知和用户偏好。
- 按任务 ID 做局部日志流或日志抽屉。
- 多项目聚合提醒和多成员在线状态。
- 数据库持久化项目事件、运行态记录和提醒历史。

# MatrixCode 第十五阶段本地执行状态轻量持久化设计规格

## 背景

第十四阶段已经让运行态提醒和已读状态通过 JSON 快照跨服务端重启恢复。浏览器重启验证也暴露了下一处体验断点：提醒记录能恢复，但提醒背后的本地工作区、任务、日志和审批审计仍然只存在于内存中。服务端重启后，用户能看到「本地命令执行成功」或「需要审批本地命令」，却看不到对应的本地执行上下文。

第十五阶段优先修补这条断点，让本地执行代理的可见历史同样支持轻量恢复。范围保持在本地 MVP：继续使用可配置 JSON 文件，不引入数据库，不恢复真实进程，不扩大到登录和多成员权限。

## 目标

- 持久化授权工作区列表，服务端重启后右侧「本地执行代理」仍能显示最近授权工作区。
- 持久化最近本地执行任务和任务日志，服务端重启后工作台仍能展示已完成、已拒绝、待审批或已取消的历史任务。
- 持久化审批审计记录，服务端重启后 `recentAuditRecords` 仍能展示最近审批链路。
- 让第十四阶段恢复的运行态提醒能继续关联到本地任务和工作区摘要。
- 存储路径可配置，默认写入 `.matrixcode/local-execution.json`，并沿用 `.matrixcode/` Git 忽略规则。

## 非目标

- 不恢复服务端重启前正在运行的 OS 进程。
- 不自动重新执行 `QUEUED` 或 `RUNNING` 任务。
- 不新增数据库、迁移脚本、ORM、Redis 或外部存储服务。
- 不持久化模型网关、文档、Bug、部署目标、Compose 环境或项目事件全量历史。
- 不新增桌面端 UI；桌面端继续读取既有 `localExecution` 工作台字段。
- 不引入用户登录、多成员权限或用户级提醒已读状态。

## 方案选择

### 方案一：用户级提醒已读

该方案沿着第十四阶段后续入口，为不同用户维护独立提醒已读状态。它适合有账号体系和多成员在线状态后再做；当前项目还没有用户身份，强行加入会让 `actorId` 和真实用户概念混在一起。

### 方案二：正式数据库持久化

该方案把提醒、任务、审计和项目事件一次性迁移到数据库表。它更接近长期形态，但会牵动依赖、迁移、测试启动和本地部署说明。当前 MVP 仍以本地可离线验证为主，直接引入数据库会扩大阶段风险。

### 方案三：本地执行状态 JSON 快照（推荐）

该方案只为本地执行代理保存可见历史，路径独立于第十四阶段提醒快照。它能立刻修复重启后提醒上下文断裂的问题，同时保持实现范围清晰。后续如果引入数据库，这份快照结构也可以作为迁移边界参考。

## 持久化范围

第十五阶段快照包含以下数据：

- `version`：当前固定为 `1`。
- `workspaces`：所有项目的 `WorkspaceAuthorization` 列表，包括已授权和已撤销记录。
- `tasks`：按项目保存 `ExecutionTask` 列表，复用 `LocalTaskStore` 现有历史裁剪规则。
- `taskLogs`：按项目和任务保存 `LocalTaskLog` 列表，复用每个任务最多 200 条日志、工作台最近 20 条日志的展示规则。
- `auditRecords`：审批与取消审计记录列表，服务端继续在摘要中做敏感信息脱敏后再写入。

第十五阶段不持久化文件操作记录和 Git diff 摘要。文件操作只是临时审计辅助，重启后缺失不会破坏提醒上下文；Git diff 可以通过现有接口重新采集，持久化旧 diff 反而容易误导用户。

## 启动恢复规则

服务端启动时从配置路径读取快照：

- 文件不存在：以空本地执行状态启动。
- 文件损坏：以空本地执行状态启动，不删除原文件。
- 版本不支持：以空本地执行状态启动。
- 已恢复的 `QUEUED` 或 `RUNNING` 任务统一转为 `CANCELED`，并追加系统日志「服务重启后任务已停止」。这样不会让用户误以为旧进程仍在运行。
- `APPROVAL_PENDING` 任务保持待审批状态，重启后仍可批准或拒绝；批准时仍会重新校验工作区是否存在且已授权。

## 架构

新增 `localexecution.persistence` 方向的轻量存储层：

- `LocalExecutionSnapshot`：可写入 JSON 的快照 record。
- `LocalExecutionStateStore`：定义加载快照、更新工作区、更新任务日志、更新审计记录的接口。
- `InMemoryLocalExecutionStateStore`：单元测试默认使用。
- `FileLocalExecutionStateStore`：Spring 环境默认使用，负责 JSON 读写、临时文件写入和原子替换。
- `LocalExecutionStorageProperties`：配置 `matrixcode.local-execution.storage-path`，默认 `.matrixcode/local-execution.json`。

各现有服务只负责自己的内存结构：

- `WorkspaceRegistry` 构造时恢复工作区列表；`authorize`、`revoke` 和访问时间更新后写回工作区分区。
- `LocalTaskStore` 构造时恢复任务和日志；`record`、`recordNew`、`replace` 和 `appendLog` 后写回任务分区。
- `AuditService` 构造时恢复审计记录；`record` 后写回审计分区。

`FileLocalExecutionStateStore` 内部维护一份当前快照，分区更新时合并其他分区后再写入文件，避免某个服务保存时覆盖其他服务已经恢复的数据。

## API 和桌面端

第十五阶段不新增 API。现有接口自然受益：

- `GET /api/projects/{projectId}/workbench`：重启后返回恢复的工作区、最近任务、最近日志和审计记录。
- `POST /api/projects/{projectId}/local-execution/commands/{taskId}/approval`：对恢复后的 `APPROVAL_PENDING` 任务仍可执行审批。
- `POST /api/projects/{projectId}/runtime-notifications/read-all`：继续只影响提醒快照，不影响本地执行快照。

桌面端不改组件和样式。只要工作台响应恢复 `localExecution` 字段，右侧现有「本地执行代理」卡片会自动展示恢复后的状态。

## 错误处理

- 写入失败时接口返回中文错误，不默默丢弃变更。
- 恢复到不存在的工作区路径时不主动删除记录；真正执行或审批时继续通过 `WorkspaceRegistry.requireAuthorized` 和文件系统检查拦截。
- 快照中的单条任务或日志无法反序列化时，文件存储按损坏文件处理，以空状态启动并保留原文件。
- 服务关闭时不额外写入状态；所有关键变更在发生时立即写回。

## 测试策略

- 文件存储测试：覆盖文件不存在、保存后加载、损坏文件容错、分区更新不互相覆盖。
- 工作区测试：服务重建后恢复授权工作区；撤销后状态也能恢复。
- 任务测试：服务重建后恢复任务和日志；恢复的运行中任务会转为取消并追加系统日志。
- 审计测试：服务重建后恢复审批审计记录。
- 控制器集成测试：使用隔离存储路径，准备待审批任务和成功任务，重建 Spring 上下文后工作台仍返回工作区、任务、日志和审计摘要。
- 全量验证继续包含服务端测试、桌面端测试、桌面端构建、Tauri 入口检查、文档扫描和浏览器重启验证。

## 验收标准

- 服务端重启后，工作台仍能显示第十五阶段前创建的授权工作区和最近本地任务。
- 第十四阶段恢复的运行态提醒对应任务在右侧「本地执行代理」里仍可追溯。
- 恢复后的待审批任务可以被拒绝；如批准，仍会重新校验工作区路径和安全策略。
- 恢复后的运行中任务不继续显示为运行中，而是显示为已取消并包含重启停止日志。
- 默认 `.matrixcode/local-execution.json` 不进入 Git 状态。
- 所有新增和既有测试通过，浏览器重启验证无控制台 error，端口和进程清理干净。

## 后续阶段入口

第十六阶段可以继续在两个方向中选择：一是把 Compose 环境、部署目标和健康检查记录纳入轻量快照，让运维运行态也能跨重启恢复；二是开始引入正式数据库边界，把提醒、本地执行和项目事件统一迁移到生产级持久化模型。

# MatrixCode 第二阶段角色工作台设计规格

## 概述

第二阶段把第一阶段的项目概览壳升级为可操作的多角色项目工作台。产品、开发、测试、运维在同一个桌面客户端内完成输入、生成、审核、冻结、流转和状态同步。服务端继续采用 Java 21 + Spring Boot 模块化单体，先使用内存仓储打通业务闭环，后续再接入 PostgreSQL、真实模型网关和本地执行代理。

本阶段不追求真正替代开发者执行所有文件修改，也不接入真实大模型。系统先提供一个可替换的本地智能体草稿生成器，用确定性规则生成中文文档草稿，让角色流转、文档版本、Bug 状态、部署目标和实时事件先跑通。

## 目标

- 让四个角色都能在平台内完成一轮有效工作动作。
- 产品可以输入需求，生成 PRD、验收标准和初始界面说明，并冻结文档版本。
- 开发可以消费冻结需求，登记本地工作区、实现说明、自测结果、接口文档、数据库脚本和部署文档。
- 测试可以创建 Bug、修改 Bug 状态、记录回归结果并生成测试报告。
- 运维可以配置环境目标、SSH 地址、部署说明、健康检查和上线准备记录。
- 所有动作都写入追加事件，并同步到项目阶段、文档交接、右侧指标和实时事件流。
- 所有文档、注释、备注和界面文案保持中文。
- 服务端验证命令继续使用 `/Users/Masons/Ai/Maven/bin/mvn` 和 `/Users/Masons/Ai/Maven_Ai_Store`。

## 非目标

- 不接入真实 OpenAI、DeepSeek、Qwen、Anthropic 或其他模型供应商。
- 不执行真实本地文件写入、命令运行、Git diff 采集或 SSH 远程操作。
- 不实现登录、组织邀请、成员权限和生产级鉴权。
- 不落 PostgreSQL 持久化，不设计数据库迁移。
- 不实现多人在线协同编辑。
- 不实现完整安装包签名、自动更新或企业分发。

## 用户流程

### 产品流程

产品进入项目后选择“产品”角色，在需求输入区描述业务目标、用户角色、功能边界和验收关注点。系统生成三类草稿：PRD、验收标准、初始界面说明。产品可以继续修改输入并重新生成草稿，也可以冻结当前版本。冻结后，项目阶段进入“需求已冻结”，开发、测试和运维可以引用该冻结版本。

### 开发流程

开发选择“开发”角色，看到冻结版需求摘要和验收标准。开发填写本地工作区路径、实现说明、自测结果、接口文档、数据库脚本和部署文档。提交后系统创建开发交付记录，并把接口文档、数据库脚本、部署文档加入文档交接区。项目阶段进入“开发交付”或“等待测试”。

### 测试流程

测试选择“测试”角色，看到冻结需求、开发交付记录和部署文档摘要。测试可以通过表单创建 Bug，填写标题、严重级别、复现步骤、期望结果、实际结果和关联角色。Bug 默认状态为“新建”。测试可以将 Bug 流转到“已确认”“修复中”“待回归”“已关闭”“重新打开”，并记录测试报告。全部关键 Bug 关闭后，项目阶段进入“待产品验收”。

### 运维流程

运维选择“运维”角色，看到开发部署文档和测试环境要求。运维填写环境名称、环境地址、SSH 地址、部署说明、健康检查地址和回滚说明。提交后系统创建环境记录。涉及 SSH 的字段只保存为待审批配置，不执行远程连接。健康检查记录和部署日志进入文档交接区，项目阶段可进入“上线准备”。

### 产品验收流程

产品在测试报告完成后进行验收。验收通过会生成验收记录，项目阶段进入“可上线”。验收不通过会写入退回原因，并将项目阶段退回开发或测试。

## 数据模型

### 项目工作台

项目工作台是桌面端主要读取的聚合视图，包含：

- 项目编号和项目名称。
- 当前阶段。
- 四个角色的任务状态、待办数量和最近动作。
- 文档列表和冻结版本。
- Bug 队列和状态计数。
- 部署目标和健康检查记录。
- 上下文指标、缓存命中率、会话 token 和事件数量。
- 最近事件流。

### 文档产物

文档产物包含：

- 文档编号。
- 项目编号。
- 文档类型：PRD、验收标准、界面说明、实现说明、接口文档、数据库脚本、部署文档、测试报告、验收记录、部署记录。
- 标题。
- 正文。
- 作者角色。
- 状态：草稿、待审核、已冻结、已替代。
- 版本号。
- 创建时间。
- 引用的上游文档编号。

冻结文档不可覆盖。重新生成或修改必须创建新版本。

### 角色动作

角色动作包含：

- 动作编号。
- 项目编号。
- 角色：产品、开发、测试、运维。
- 动作类型：生成草稿、冻结文档、提交开发交付、创建 Bug、流转 Bug、提交测试报告、配置部署目标、提交验收。
- 输入摘要。
- 生成或引用的文档编号。
- 结果状态。
- 事件时间。

### Bug

Bug 包含：

- Bug 编号。
- 项目编号。
- 标题。
- 严重级别：低、中、高、阻塞。
- 状态：新建、已确认、修复中、待回归、已关闭、重新打开。
- 复现步骤。
- 期望结果。
- 实际结果。
- 关联文档编号。
- 创建角色。
- 当前处理角色。
- 最近更新时间。

### 部署目标

部署目标包含：

- 目标编号。
- 项目编号。
- 环境名称。
- 环境地址。
- SSH 地址。
- 部署说明。
- 健康检查地址。
- 回滚说明。
- 状态：待配置、待审批、已记录、上线准备、部署完成。

## 服务端设计

### 模块边界

- `workbench`：项目工作台聚合视图、角色动作入口、阶段投影。
- `agent`：本地智能体草稿生成器，定义可替换接口，不绑定真实模型。
- `document`：沿用第一阶段文档版本能力，补充文档查询和冻结入口。
- `workflow`：沿用第一阶段状态事件能力，补充角色动作到项目阶段的投影。
- `bug`：新增 Bug 创建、查询和状态流转。
- `deployment`：新增部署目标配置和环境记录。
- `realtime`：沿用 SSE 事件流，角色动作完成后发布项目事件。
- `usage` 和 `context`：继续输出缓存命中率、token 和上下文产物列表，第二阶段先使用估算值。

### API 草案

- `GET /api/projects/{projectId}/workbench`：读取完整项目工作台。
- `POST /api/projects/{projectId}/roles/product/drafts`：根据产品输入生成文档草稿。
- `POST /api/projects/{projectId}/documents/{documentId}/freeze`：冻结文档版本。
- `POST /api/projects/{projectId}/roles/developer/deliveries`：提交开发交付。
- `POST /api/projects/{projectId}/bugs`：创建 Bug。
- `POST /api/projects/{projectId}/bugs/{bugId}/transitions`：流转 Bug 状态。
- `POST /api/projects/{projectId}/roles/tester/reports`：提交测试报告。
- `POST /api/projects/{projectId}/deployments/targets`：配置部署目标。
- `POST /api/projects/{projectId}/roles/product/acceptance`：提交产品验收。
- `GET /api/projects/{projectId}/events/stream`：继续使用第一阶段 SSE 实时事件接口。

### 本地智能体草稿生成器

本地智能体草稿生成器负责把用户输入转换为稳定、可测试的中文结构化草稿。它不是聊天模型，也不模拟推理过程。它必须满足：

- 相同输入得到稳定输出，便于测试。
- 输出包括标题、摘要、正文和验收关注点。
- 生成内容要明确标注为“草稿”，必须由对应角色审核或冻结后才能流转。
- 通过接口抽象，后续可以替换为真实模型网关。

## 桌面端设计

### 布局

保持第一阶段三栏工作台：

- 左侧栏：项目、角色切换、当前阶段、待办数量。
- 中间区：当前角色工作区，显示该角色可执行动作和最近产物。
- 右侧栏：上下文指标、文档交接、Bug 队列、部署状态、事件流。

界面继续保持深色、紧凑、工作台风格，不做营销式首页，不使用大面积装饰图形。

### 角色工作区

产品工作区包含需求输入、草稿生成、草稿预览和冻结按钮。

开发工作区包含工作区路径、实现说明、自测结果、接口文档、数据库脚本、部署文档和提交按钮。

测试工作区包含 Bug 表单、Bug 状态选择、回归说明、测试报告和提交按钮。

运维工作区包含环境名称、环境地址、SSH 地址、部署说明、健康检查地址、回滚说明和提交按钮。

### 实时反馈

每次提交动作后：

- 当前角色区域显示提交结果。
- 阶段条更新。
- 右侧事件流追加新事件。
- 文档交接列表刷新。
- Bug 队列或部署目标刷新。
- 缓存命中率、token 和上下文产物列表刷新。

## 状态与事件

所有动作都发布项目事件。事件类型包括：

- `PRODUCT_DRAFT_CREATED`
- `DOCUMENT_FROZEN`
- `DEVELOPER_DELIVERY_SUBMITTED`
- `BUG_CREATED`
- `BUG_TRANSITIONED`
- `TEST_REPORT_SUBMITTED`
- `DEPLOYMENT_TARGET_CONFIGURED`
- `ACCEPTANCE_SUBMITTED`
- `WORKFLOW_STAGE_CHANGED`

当前阶段由事件投影产生，优先级如下：

1. 没有冻结需求：需求草稿。
2. 有冻结需求但没有开发交付：开发中。
3. 有开发交付但没有测试报告：测试中。
4. 有阻塞或高严重级别未关闭 Bug：缺陷处理中。
5. 有测试报告但没有产品验收：待产品验收。
6. 验收通过且已有部署目标：上线准备。
7. 验收通过但没有部署目标：待运维配置。

## 权限与安全

第二阶段不做完整身份系统，但接口层仍保留角色动作边界：

- 产品接口只处理产品输入、冻结和验收。
- 开发接口只处理开发交付记录。
- 测试接口只处理 Bug 和测试报告。
- 运维接口只处理部署目标和环境记录。
- SSH 地址只作为配置文本保存，不触发连接。
- 工作区路径只作为开发交付元数据保存，不读取本地文件。
- 所有动作写入审计摘要，并通过实时事件展示。

## 测试策略

服务端使用 JUnit 5：

- 本地智能体草稿生成器测试：同一输入稳定输出中文草稿。
- 工作台服务测试：产品生成草稿后工作台出现文档。
- 文档冻结测试：冻结后状态和版本正确。
- 开发交付测试：提交后文档交接和阶段更新。
- Bug 测试：创建和状态流转正确。
- 部署目标测试：配置后状态更新且不执行 SSH。
- 工作台接口测试：返回项目、角色、文档、Bug、部署和指标。

桌面端使用 Vitest：

- 客户端 API 测试：覆盖工作台读取和角色动作提交。
- 角色切换测试：四个角色工作区都能显示。
- 产品流程测试：输入需求、生成草稿、冻结文档。
- 测试流程测试：创建 Bug 并看到状态更新。
- 运维流程测试：提交环境信息后看到部署记录。

## 验收标准

- 浏览器打开桌面端后，能看到四个角色入口。
- 切换到产品角色，输入需求后能生成中文 PRD、验收标准和界面说明草稿。
- 产品可以冻结草稿，冻结后阶段条进入开发相关阶段。
- 切换到开发角色，能看到冻结需求，并提交开发交付产物。
- 切换到测试角色，能创建 Bug、流转 Bug 状态，并提交测试报告。
- 切换到运维角色，能配置环境目标、SSH 地址和健康检查信息，但不会执行远程连接。
- 右侧文档交接、Bug 队列、部署状态和事件流会随动作更新。
- 服务端测试通过，命令使用 `/Users/Masons/Ai/Maven/bin/mvn -Dmaven.repo.local=/Users/Masons/Ai/Maven_Ai_Store -q -pl server test`。
- 桌面端测试和构建通过。

## 后续阶段入口

第三阶段建议接入真实模型网关和缓存优化：

- 模型供应商配置。
- 角色模型选择。
- 稳定提示词前缀和工具契约。
- 请求上下文清单。
- prompt cache 命中统计。
- token 成本和压缩策略。

第四阶段建议接入真实本地执行代理：

- 工作区授权。
- 文件读写。
- 命令执行。
- Git diff 采集。
- SSH 审批与部署动作。
